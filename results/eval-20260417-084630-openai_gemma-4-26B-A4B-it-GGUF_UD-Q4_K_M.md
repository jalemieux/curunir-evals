To evaluate the transition from the current `curunir` architecture to a "Hermes" framework, I have analyzed the existing implementation against the proposed components (LangGraph, Pydantic, and Temporal).

### Architectural Comparison

| Feature | **curunir** (Current) | **Hermes** (Proposed) |
| :--- | :--- | :--- |
| **Control Flow** | **Loop-based:** Imperative `while` loop with a fixed iteration cap. Linear history management. | **Graph-based:** Declarative state machines. Nodes and edges define transitions, allowing for complex cyclic flows and conditional branching. |
| **Data Integrity** | **Implicit/Loose:** Tool arguments are handled via LLM generation; type safety depends on tool implementation. | **Strictly Typed:** Pydantic ensures every tool call adheres to a schema before execution, reducing "hallucinated" arguments. |
| **Reliability** | **Transient:** If the process crashes mid-task, the state is lost. Recovery requires manual intervention or re-running. | **Durable:** Temporal provides event sourcing. If the system crashes, it resumes exactly where it left off, preserving state and history. |

---

### The Trade-off Analysis

#### What we would gain:
1.  **Reliability at Scale:** Temporal allows for long-running tasks (days/weeks) that survive restarts.
2.  **Complex Logic:** LangGraph enables multi-agent choreography that isn't just "orchestrator $\to$ specialist" but rather a collaborative graph.
3.  **Robustness:** Pydantic reduces the "trial and error" loop currently spent on fixing malformed tool calls.

#### What we would break:
1.  **Latency & Overhead:** The current lightweight `asyncio` loop is extremely fast. Adding a state machine and a distributed scheduler (Temporal) introduces significant orchestration latency.
2.  **Complexity/Cognitive Load:** `curunir` is currently easy to debug by following a single execution thread. A distributed, stateful graph is significantly harder to trace.
3.  **Development Velocity:** Adding a new tool in `curunir` is a simple function; in Hermes, it requires defining Pydantic schemas, graph nodes, and potentially Temporal workflows.

---

### Proposed Enhancements

I propose a phased approach to bridge the gap without destroying the current speed and simplicity.

| Enhancement | Description | Priority | Effort |
| :--- | :--- | :--- | :--- |
| **1. Pydantic Tool Registry** | Refactor existing tool definitions to use Pydantic models for input validation. This provides the "Hermes" safety without the graph complexity. | **P0** | Low |
| **2. State-Checkpointing** | Implement a lightweight "snapshot" mechanism in `Agent.handle()` that saves current history and pending tool calls to `context/memory/` after every tool execution. | **P0** | Med |
| **3. DAG-based Skills** | Instead of a single `Agent`, allow `Skills` to define a simple Directed Acyclic Graph (DAG) of sub-tasks, moving closer to LangGraph's modularity. | **P1** | High |
| **4. Temporal Integration** | Only for the `scheduler` component. Offload long-running or high-reliability cron tasks to a Temporal worker to ensure they actually execute. | **P1** | High |
| **5. Full Graph Orchestration** | Replace the `orchestrator` loop with a formal LangGraph-style state machine for complex, multi-step multi-agent workflows. | **P2** | Very High |

2.  **Ingestion**: Once a new message is identified, `EmailChannel` wraps the content into an `IncomingMessage` object (defined in `src/channels/base.py`) and pushes it into an `asyncio.Queue` via `self.in_queue.put()`.
